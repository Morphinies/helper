# Архитектура проекта

Проект построен на Next.js App Router. Основной код находится в `src`.

## Основные технологии

- `next` - роутинг, layouts, metadata и серверный рендеринг.
- `next-intl` - локализация и локализованный роутинг.
- `next-auth` - авторизация.
- `antd` - UI-компоненты.
- `@tanstack/react-query` - клиентские async-запросы.
- `@reduxjs/toolkit` и `react-redux` - глобальное клиентское состояние.
- `sass` - стили модулей и глобальные стили.

## Структура `src`

- `app` - маршруты Next.js, layouts, providers и API route handlers.
- `features` - пользовательские сценарии и действия.
- `entities` - бизнес-сущности приложения.
- `widgets` - крупные UI-блоки и экраны приложения.
- `shared` - переиспользуемые утилиты, UI-компоненты, типы и инфраструктурный код.
- `i18n` - настройки локализации, навигации и загрузки переводов.
- `messages` - JSON-файлы переводов.
- `styles` - глобальные стили, шрифты и настройки темы.

## Роутинг

Все пользовательские страницы находятся внутри динамического сегмента
`src/app/[locale]`.

Локали описаны в `src/i18n/routing.ts`:

- доступные локали: `en`, `ru`;
- дефолтная локаль: `en`;
- `localePrefix: "as-needed"`.

Это значит, что для английской локали URL идут без префикса:

- `/`
- `/tasks`
- `/login`

Для русской локали используется префикс:

- `/ru`
- `/ru/tasks`
- `/ru/login`

Для разделения layouts используются route groups:

- `src/app/[locale]/(app)` - основные страницы приложения с header.
- `src/app/[locale]/(auth)` - страницы авторизации без основного header.

Route groups не добавляют сегмент в URL.

## Layouts и providers

`src/app/[locale]/layout.tsx` - корневой locale layout. Он отвечает за:

- подключение глобальных стилей;
- проверку валидности locale;
- установку request locale для `next-intl`;
- `html lang`;
- подключение `NextIntlClientProvider`;
- подключение общих providers.

Общие providers находятся в `src/app/providers/index.tsx`:

- `SessionProvider`;
- Redux `Provider`;
- `QueryClientProvider`;
- `AntdRegistry`;
- `ThemeProvider`.

Layout основных страниц находится в `src/app/[locale]/(app)/layout.tsx` и
оборачивает страницы в `widgets/Layout`.

Layout auth-страниц находится в `src/app/[locale]/(auth)/layout.tsx` и не
рендерит основной header.

## Features

`features` содержит пользовательские сценарии и действия, которые могут
использоваться внутри страниц и widgets.

Текущие features:

- `auth` - кнопка входа/выхода и форма логина.

Внутри feature используется локальная структура:

- `ui` - React-компоненты.
- `model` - hooks, типы и состояние feature, если они нужны.
- `lib` - чистые helper-функции, если они нужны.
- `index.ts` - публичный экспорт feature.

## Entities

`entities` содержит бизнес-сущности приложения: типы, модель, storage и чистые
helper-функции, которые описывают предметную область, а не конкретный экран.

Текущие entities:

- `task` - типы задачи, работа со списком задач, local storage и helper-функции
  для статусов и формы задачи.

Внутри entity используется локальная структура:

- `model` - типы, hooks и работа с данными сущности.
- `lib` - чистые helper-функции сущности.
- `index.ts` - публичный экспорт entity.

## Widgets

`widgets` содержит самодостаточные крупные части интерфейса.

Текущие widgets:

- `Layout` - основной layout приложения с меню, переключателем locale, темой и
  auth-кнопкой из `features/auth`.
- `PageWrapper` - серверная защита страницы, загрузка переводов страницы и
  клиентская обёртка контента.
- `Habits` - экран привычек, его UI, model, lib и local storage.
- `Tasks` - экран задач, его UI и экранная model. Данные и helpers задачи
  находятся в `entities/task`.

Внутри widget используется локальная структура:

- `ui` - React-компоненты и CSS modules.
- `model` - hooks, типы и состояние конкретного widget.
- `lib` - чистые helper-функции.
- `index.ts` - публичный экспорт widget.

## Shared

`shared` содержит код, который не принадлежит конкретному экрану:

- `shared/ui` - переиспользуемые UI-компоненты.
- `shared/lib` - инфраструктурные настройки, например store и auth options.
- `shared/utils` - общие утилиты.
- `shared/types` - общие типы.

Новый код стоит класть в `shared` только если он реально переиспользуется между
несколькими частями приложения.

## Локализация

Настройки локализованного роутинга находятся в `src/i18n/routing.ts`.

Локализованные helpers для навигации находятся в `src/i18n/navigation.ts`.

Загрузка сообщений находится в `src/i18n/messages.ts`:

- `common` используется для общих переводов;
- `habits` используется для страницы привычек;
- `tasks` используется для страницы задач.

Файлы переводов лежат в `src/messages/{locale}`.

Для page-specific переводов страница передаёт нужный `messagesKey` в
`PageWrapper`. Metadata страницы берётся из JSON соответствующей страницы.

## Стили

Глобальные стили находятся в `src/styles/globals.scss`.

Компонентные стили пишутся через CSS modules рядом с компонентами:

- `Component.module.scss`
- импорт как `s`;
- классы используются через `s["class-name"]`.

Настройки темы находятся в `src/styles/themeConfig.ts`, а переключение темы
проходит через `ThemeProvider` и `ThemeButton`.

## Правила добавления новых страниц

Для новой обычной страницы приложения:

1. Добавить route в `src/app/[locale]/(app)`.
2. Создать или переиспользовать widget для основной UI-логики.
3. Создать или переиспользовать feature для пользовательского сценария, если
   логика не является частью конкретного экрана.
4. Создать или переиспользовать entity для бизнес-сущности, если логика
   описывает предметную область.
5. Добавить JSON переводов в `src/messages/en` и `src/messages/ru`.
6. Добавить namespace в `src/i18n/messages.ts`, если страница требует отдельный
   JSON.
7. Передать нужный `messagesKey` в `PageWrapper`.
8. Metadata страницы хранить в JSON соответствующей страницы.

Для новой auth-страницы:

1. Добавить route в `src/app/[locale]/(auth)`.
2. Не использовать основной `widgets/Layout`.
3. При необходимости проверять сессию на сервере через `getServerSession`.

## Code style

При изменении кода нужно соблюдать правила из `AGENTS.md`:

- предпочитать визуально аккуратный порядок механическому алфавитному;
- в named imports держать простые imports раньше, `type` imports - в конце;
- не переставлять несвязанный код только ради сортировки;
- в props и object types сначала перечислять data-поля, затем callbacks и
  getter-функции;
- в JSX props сначала указывать стабильные data props, затем поведение и
  handlers рядом с соответствующей логикой.
